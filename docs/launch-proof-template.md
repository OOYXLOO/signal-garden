# Signal Garden Launch Proof Template

This template turns a technical demo into a reviewer-ready launch proof pack. It is designed for public evidence only: playable URLs, source links, screenshots, video, and copyable submission fields.

## 1. Audience And First-Minute Promise

- Primary reviewer: a judge, platform reviewer, or community moderator who has less than one minute to decide whether the project is real.
- First-minute promise: the reviewer can open one public desk, launch the app, load a complete sample route, inspect source evidence, and understand the community loop without account access.
- Required proof shape: every claim should point to a public URL, a repository file, or a deterministic local command.

## 2. Reviewer Fast Path

1. Open the public judge desk.
2. Click the playable app link.
3. Load the current sample route with `?sample=1`.
4. Confirm that the board, route replay, beacon progress, and receiver state are visible.
5. Open the public proof checklist.
6. Open the source repository and submission manifest.
7. Watch the short captioned demo only if the interactive path is not enough.

## 3. Public Proof Checklist

- Public app URL works without sign-in.
- Judge desk links to app, source, media, checklist, manifest, and field packs.
- Sample route URL opens a solved or reviewable path for the current day.
- Source repository contains the relevant game loop, evidence scripts, and audit commands.
- Submission manifest lists evidence file byte counts and SHA-256 hashes.
- Screenshot set covers desktop, mobile, and an external-facing cover image.
- Short demo video shows the board, route, community import loop, and submission readiness state.
- Community loop is understandable from the page or docs without a private platform account.
- Return loop is visible through the archive, streak, pledge, or daily progression surface.
- Remaining platform gates are named as pending public URLs, not hidden inside private account pages.

## 4. Screenshot And Video Shot List

- Cover shot: project name, board state, and core visual hook.
- Desktop shot: full board with reviewer panels visible.
- Mobile shot: touch-sized board and readable status panels.
- Community shot: sample comments or review links imported into ranked proposals.
- Return shot: seven-day archive, streak, pledge, or daily return prompt.
- Submission shot: readiness checklist or launch packet panel.
- Video opening: public app URL and sample route load.
- Video middle: route replay, top proposal, and evidence receipt.
- Video ending: judge desk, source repository, and public checklist.

## 5. Submission Field Handoff

Prepare these fields as copyable public text:

- Project name.
- Short tagline.
- Public app URL.
- Source repository URL.
- Judge desk URL.
- Demo video URL or repository media path.
- What it does.
- How it uses the platform or community mechanic.
- What makes it replayable or return-worthy.
- Technical implementation notes.
- Testing instructions.
- Known limitations and pending external gates.

## 6. Missing Evidence And Risk Notes

Use this section before any final submission:

- If an app listing is blocked by an account or humanity check, write the intended listing URL placeholder and keep the public web build as the review path.
- If a demo post is not yet public, use the repository draft and mark the post URL as pending.
- If a platform survey or form is required, keep field answers in the repository but submit only through the official form.
- If a link depends on GitHub Pages, run the public URL audit after the Pages deployment completes.
- If any media shows private account data, regenerate it before publishing.
